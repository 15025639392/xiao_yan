# xiao_yan 测试策略

本文档用于定义 `xiao_yan` 在高变化阶段的测试策略。

目标不是尽快把覆盖率做高，而是在项目仍快速演进时，用少量高价值测试守住主链路、数字人本体边界、可选依赖降级能力，以及后续重构的安全性。

本策略默认适用于整个仓库，重点适用于：

- `services/core`
- `apps/desktop`
- `tools/skills`

## 一、核心结论

`xiao_yan` 不适合把测试整体推迟到“项目后期”再统一补。

原因不是“越早越工程化越好”，而是这个项目的主要风险并不只是普通功能回归，还包括：

- 主链路被改断，但局部代码看起来还能运行
- 数字人本体被工具化，行为仍可执行但方向跑偏
- 可选依赖逐渐变成硬依赖
- 大文件继续堆逻辑，后续拆分缺少行为护栏
- bug 被修一次又回归一次

因此，本仓库默认采用：

- 前期少量高价值测试
- 中期按稳定边界逐步补强
- 后期再补更细粒度和更高维护成本的测试

## 二、当前阶段的测试优先级

### 必须优先覆盖

- 后端 API 契约与主流程冒烟
- persona / memory / goals / world 的稳定行为规则
- 可选依赖缺失时的降级行为
- 修复过的真实 bug 回归测试
- 大文件拆分前的外部行为测试
- 前端关键页面的加载、关键数据请求与关键操作测试

### 可以延后覆盖

- 频繁变化的 UI 细节断言
- 对内部实现绑定很深的 mock-heavy 单测
- 仍在高频摇摆的长链路端到端脚本
- 视觉结构、样式层级、组件实现方式等易碎断言

## 三、按仓库模块划分的测试策略

### 1. `services/core`

后端是当前项目的主稳定面，测试投入应最高。

前期优先测：

- `api -> service` 级别的契约与主流程
- 领域行为规则，而不是实现细节
- 可选依赖降级与 fallback
- 权限、审批、sandbox、capability 边界

优先参考现有测试分布：

- API 契约：`services/core/tests/test_api_*.py`
- 目标与自治：`services/core/tests/test_goal_*.py`、`services/core/tests/test_autonomy*.py`
- 记忆：`services/core/tests/test_memory_*.py`
- 人格：`services/core/tests/test_persona*.py`、`services/core/tests/test_expression_*.py`
- 工具与能力：`services/core/tests/test_tools_*.py`、`services/core/tests/test_capability_*.py`
- 可选依赖边界：`services/core/tests/test_mempalace_adapter.py`

当前仓库内值得优先守住的高风险行为：

- chat 请求在主链路内仍能正确组装 persona、memory、world、goals 信息
- goal admission 不会在重构中改坏状态流转
- memory 缺少可选依赖时仍可清晰降级
- tool / capability 审批与执行边界不被放宽
- runtime config 与启动流程不把可选依赖升格为默认硬前置

### 2. `apps/desktop`

桌面端前期不追求细碎组件测试，优先守住页面级行为。

前期优先测：

- 页面挂载后是否请求关键数据
- 关键 tab / mode 切换是否触发正确请求
- 关键按钮是否发出正确动作
- 关键空态、错误态、审批态是否仍能表达

不建议前期重投入：

- DOM 结构细节
- 样式类名细节
- 频繁变化的布局顺序

当前仓库内合适的测试形态已经存在，例如：

- `apps/desktop/src/pages/MemoryPage.test.tsx`
- `apps/desktop/src/pages/CapabilitiesPage.test.tsx`

这类测试价值高，因为它们测的是“页面是否仍在正确地理解后端接口与用户操作”，而不是页面内部如何组织实现。

### 3. `tools/skills`

项目 skill 不是普通说明文档，它们实际承担“约束 AI 如何进入问题、如何收尾、哪些风险必须显式处理”的作用。

因此，skill 的“测试”不一定都表现为自动化测试文件，也可以表现为：

- 明确的触发条件
- 明确的必产出项
- 明确的门禁和否决条件
- 与代码改动收尾动作的同步约束

对 skill 这层，前期优先保证：

- 文档规则不互相冲突
- 推荐顺序与实际仓库结构匹配
- 关键 skill 有清晰的测试义务说明
- 改完代码后，skill 对应的验证动作没有失效

如果后续要加强，可考虑再补脚本化校验，例如：

- 检查 `docs/skills.md` 中列出的 skill 是否都真实存在
- 检查 skill 文档中的关键引用路径是否失效
- 检查 project skill 是否仍符合当前仓库目录结构

## 四、项目专用测试义务矩阵

下面这组规则用于把“什么时候必须补测试”从经验变成默认纪律。

### `xiao-yan-architecture-guardian`

适用范围：

- memory / goals / persona / tools / safety / runtime flow / 长期状态 / 权限边界

最低测试义务：

- 至少确认这轮改动的外部行为验证点存在
- 如果改动进入主链路，至少补一个主流程冒烟测试或行为断言
- 不要求优先补深度实现单测，优先补能保护主体性、连续性、边界感的行为测试

### `core-change-entrypoint`

适用范围：

- 修改 `services/core` 入口、主链、route、service、repository、状态模型

最低测试义务：

- 在动代码前确认该链路测试覆盖状态：已有 / 部分 / 没有
- 如果是“没有覆盖”的入口改动，本轮至少补一个最小冒烟测试
- 如果目标文件已超预算，优先补外部行为测试再做结构调整

### `runtime-bug-triage`

适用范围：

- 运行时行为异常、链路异常、状态错位、显示与数据不一致

最低测试义务：

- 每个被定位并修复的真实 bug，都应补至少一个回归断言
- 回归断言优先覆盖这次真实失效分支，不追求顺手扩大覆盖面

这是当前仓库最应该长期坚持的默认测试纪律。

### `large-file-split-advisor`

适用范围：

- 超预算文件、长函数、混合职责模块

最低测试义务：

- 先锁外部行为，再拆内部结构
- 如果没有现成行为测试，本轮拆分前至少补一个主行为测试
- 拆分后的测试应继续贴近外部行为，不要跟着内部重排一起变脆

### `optional-dependency-boundary-check`

适用范围：

- `mempalace`、`chromadb`、`pypdf`、adapter、repository、导入链、降级逻辑

最低测试义务：

- 至少验证缺失依赖时系统仍可启动或可清晰降级
- 至少验证错误提示、fallback 或关闭路径其中一项
- 如果边界有调整，应补一条防止“可选依赖硬依赖化”的测试

这是 `xiao_yan` 区别于普通项目的一类高价值测试，优先级很高。

### `docs-and-tests-sync-guard`

适用范围：

- 任何会改变“别人如何理解或验证系统”的改动

最低测试义务：

- 最终说明中必须列出实际跑过的测试或检查
- 没跑的测试要明确未验证链路和剩余风险
- 如果改动影响 skill、依赖、配置、状态结构、API、降级行为，要同步检查文档与测试说明是否过期

### `project-simplifier`

适用范围：

- 项目收敛、做减法、冻结模块、删除候选、压缩主链路

最低测试义务：

- 优先保住主链路冒烟测试
- 删除模块时同步删除或更新失效测试
- 做减法后至少验证当前默认入口、默认页面和默认 API 仍可成立

## 五、按阶段执行的测试策略

### 前期

目标：

- 守住主链路
- 守住边界
- 降低返工成本

执行方式：

- 不追覆盖率
- 只补高价值测试
- bug 修复必须补回归
- 大文件拆分先测后拆

### 中期

目标：

- 围绕已经稳定的模块边界补齐测试

执行方式：

- 增加 `route -> service -> repository` 集成测试
- 增加配置、降级、异常分支测试
- 对热点模块逐步减少无测试重构面

### 后期

目标：

- 提升长期演进信心和回归效率

执行方式：

- 补更完整的前端交互测试
- 补更完整的端到端脚本
- 补更细粒度的内部单测

## 六、当前仓库建议的提交前检查

如果希望直接复用一份收尾模板，可使用 `docs/test-checklist-template.md`。

如果本轮改动涉及后端主链、状态结构、可选依赖、权限边界、skill 或大文件拆分，建议至少执行以下检查中的一部分：

```bash
cd /Users/ldy/Desktop/work/xiao_yan/services/core && pytest
cd /Users/ldy/Desktop/work/xiao_yan/apps/desktop && npm test
cd /Users/ldy/Desktop/work/xiao_yan && python3 tools/check_file_budgets.py
```

如果不是全量验证，至少执行与本轮改动直接相关的测试子集，并在结果说明里明确：

- 跑了什么
- 没跑什么
- 剩余风险是什么

## 七、默认决策规则

当你不确定当前要不要补测试时，默认按下面判断：

- 会影响主链路：补
- 会影响 API 或状态契约：补
- 会影响可选依赖边界或降级行为：补
- 会修复真实 bug：补
- 会拆大文件或重排热点逻辑：先补
- 只是频繁变化的 UI 细节：可延后
- 只是实现细节重排且已有外部行为护栏：可不补新的细粒度测试

## 八、收尾要求

任何完成的代码改动，在汇报时至少回答：

- 大文件压力是变好、持平，还是变差
- 重复逻辑是减少、持平，还是增加
- 是否触达性能敏感路径
- 跑了哪些测试和检查
- 没跑哪些验证，剩余风险是什么

这不是额外负担，而是为了让后续人类和 AI 都能继续稳定接手。
