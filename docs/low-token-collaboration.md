# xiao_yan 低 Token 协作规则

本文档用于约束 `xiao_yan` 在项目初期的低 token 协作方式。

目标不是减少必要思考，而是减少无效扫描、重复定位、过宽任务和过长上下文。

当前项目仍处于高变化阶段，默认优先：

- 小任务
- 窄范围
- 短链路
- 相关测试子集
- 必要 skill，避免全套连用

## 一、默认工作方式

### 1. 一次只打一条行为链

优先把任务压成一条最小链路，例如：

- 只找入口
- 只补回归测试
- 只修一个映射错误
- 只拆一个纯逻辑块

不推荐把这些混成一个大请求：

- 全仓分析
- 架构评审
- 编码
- 补测试
- 补文档
- review

除非用户明确要求，否则不要默认把它们绑成同一轮。

### 2. 默认限制范围

任务描述最好显式包含：

- `目标`
- `范围`
- `不要做`
- `产出`
- `验证`

如果用户没有给范围，AI 也应先尽量把范围收窄到：

- 单模块
- 单文件
- 单链路
- 单测试文件

### 3. 默认先看目标文件，再补最小上下文

推荐顺序：

1. 目标文件
2. 相关测试
3. 直接依赖链
4. 必要规范文档

不推荐每次都先扫：

- 全部 `docs/`
- 全部 `plans/`
- 全部 `tools/skills/`

### 4. 默认只跑相关验证

除非用户要求全量验证，否则优先：

- 相关 `pytest` 子集
- 相关 `vitest` 子集
- 必要的 `python3 tools/check_file_budgets.py`

不默认全量跑整仓测试。

### 5. 默认少触发重 skill

前期推荐：

- 局部 bug：优先 `runtime-bug-triage`
- 局部后端改动：优先 `core-change-entrypoint`
- 热点大文件：必要时再加 `large-file-split-advisor`
- 主流程 / 边界 / 本体问题：才优先 `xiao-yan-architecture-guardian`
- 项目收敛：才优先 `project-simplifier`

## 二、推荐请求模板

建议优先使用下面这个模板发起任务：

```text
目标：<一句话目标>
范围：<只允许看的文件、模块或链路>
不要做：<明确排除项>
产出：<只分析 / 直接改代码 / 只补测试 / 只出方案>
验证：<只跑哪些测试或不跑>
```

### 示例 1

```text
目标：给 goals admission 补拆分前护栏测试
范围：只看 services/core/app/goals/admission.py 和相关 tests
不要做：不要拆文件，不要改 runtime_config，不要补文档
产出：直接改代码并跑相关 pytest
验证：只跑 services/core/tests/test_goal_admission.py
```

### 示例 2

```text
目标：定位 chat 返回状态和前端显示不一致的问题
范围：只看相关 route、response mapper 和对应 tests
不要做：不要顺手重构，不要扫全仓
产出：先分析根因和最小修复面
验证：先不跑全量测试
```

## 三、对 AI 的默认约束

- 不默认做全仓扫描
- 不默认把多个 skill 串成长链
- 不默认读长文档，只读和当前任务直接相关的最小入口
- 不默认输出大而全方案，优先给最小闭环
- 不默认全量跑测试
- 不默认把文档、重构、测试和功能改动绑成超大任务

## 四、适用边界

以下情况可以放宽低 token 约束：

- 重要架构决策
- 主流程重排
- 可选依赖边界调整
- 默认入口、状态结构、权限边界变化
- 用户明确要求全局调研或系统性方案
