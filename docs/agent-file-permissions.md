# Agent 自动改源码: 权限与 UI 推荐方案

目标: 让 `services/core` 里的 agent 可以自动读取/修改本机源码目录, 同时让权限边界清晰且不容易误操作.

## 结论

- 如果采用 **core 的 agent 自动改(后端工具链写文件)**, 那么目录授权应该属于 **后端 Agent 的能力边界**, 不应放在 Chat 的常规设置里.
- 目录路径应尽量 **写死或构建期固定**, 避免在 UI 上随意授权任意目录导致安全风险.

## 权限模型

### 1) 后端 Agent 文件权限 (推荐保留, 适配自动改代码)

- 生效位置: `services/core` (后端)
- 作用对象: agent 使用的文件工具(如 file tools)
- 权限含义: 允许 agent 在哪些目录进行读取/写入, 以及读写级别
- 用途: "自动改源码" 的关键开关

建议:

- 默认不授权任何目录
- 只提供对 "项目根目录" 的一键授权(读写), 并且这个路径应通过配置固定(不让用户自由输入路径)
- UI 文案明确: 这是给 **后端 Agent** 的权限, 不是本地 App 自己读写文件的权限

### 2) Tauri 本地文件系统沙箱 (可选, 用于本地壳/前端读写)

- 生效位置: `apps/desktop` 的 Tauri 壳 (本机)
- 作用对象: Tauri 后端命令(例如 `fs_read_text_file` / `fs_write_text_file`)
- 权限含义: 仅允许访问用户选择的根目录内的相对路径, 禁止绝对路径与 `..` 越权
- 用途: 本地 App 自己的文件能力(例如本地浏览/编辑), 对 "core agent 自动改" **不直接生效**

建议:

- 如果主要使用后端 agent 自动改代码, 可将该入口隐藏到开发者区或单独页面, 避免用户误以为它控制了后端 agent 的权限.

## UI 放置建议

- 不建议: 放在 Chat 设置(容易让用户误解为聊天参数, 也会让 Chat 配置面板越来越臃肿)
- 推荐: 放在 "工具箱/系统/Agent" 类页面
  - 例如新增一个 `🤖 Agent` tab, 标题 "后端 Agent 文件权限"
  - 提供 "一键授权项目目录(读写)" + 当前授权列表 + 移除按钮

## 固定路径方案

推荐优先级:

1. 构建期注入固定路径(推荐):
   - 例如使用 `VITE_AGENT_PROJECT_ROOT=/abs/path/to/repo`
   - 好处: 打包/部署时可以锁死路径, 同时允许不同机器不同路径
2. 后端配置文件固定路径:
   - 在 `services/core` 里通过配置项指定允许目录白名单
3. 代码写死路径(不推荐):
   - 不利于跨机器/CI/发布, 只能用于临时调试

## 后续可讨论的演进方向

- 统一权限来源: 是否需要把后端的文件访问也代理到 Tauri, 由本地壳统一把关
- 多项目支持: 如果以后要支持多个仓库, 如何管理多个 "项目根目录" 授权
- 生产安全: 如何在发布版本里限制只允许特定根目录, 防止被误授权到敏感路径
