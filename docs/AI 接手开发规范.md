# AI 接手开发规范

## 目的

本文档用于约束 AI 在 `xiao_yan` 项目中的接手、续写、重构与修复行为。

目标不是让 AI “尽量多做事”，而是让 AI 在理解项目目标、架构边界和依赖策略的前提下，稳定地做对事、少做错事、少引入长期维护负担。

本文档默认适用于整个仓库，重点适用于 `services/core`。

## 核心目标

- 优先保护“小晏”作为持续存在数字人的主体性、连续性和边界感。
- 优先保持代码可读、可追踪、可回退，而不是追求表面上的先进或复杂。
- 优先让后续 AI 和人类都能快速接手，而不是制造只对当前修改者友好的局部优化。

## 一、总原则

### 1. 显式优先于隐式

- 优先使用普通 Python 模块、明确函数调用和清晰数据流。
- 避免依赖大量框架魔法、运行时注册、隐式注入和难追踪的全局副作用。
- 一个改动应尽量让人通过 `rg` 和目录结构就能追到主要行为。

### 2. 少依赖优先于多依赖

- 新功能默认先用标准库和现有依赖实现。
- 只有在现有手段明显不够、并且收益长期成立时，才考虑新增第三方库。
- 引入新库前必须先回答：
  - 不用它能不能做。
  - 它会不会引入新的隐式行为或复杂初始化链路。
  - 它是否会降低后续 AI 的接手效率。

### 3. 小步修改优先于大重写

- 优先做局部、可验证、可回滚的变更。
- 没有明确收益时，不做跨模块的大范围重排。
- 已有接口如果仍然清楚可用，优先在现有结构上扩展，而不是先重构再实现。

### 4. 边界优先于技巧

- 先保证模块职责清楚，再考虑抽象是否优雅。
- 避免为了“看起来更通用”而提前引入复杂抽象层。
- 如果一个抽象没有同时减少重复、降低耦合、提升可读性，就不应引入。

### 5. 体积受控优先于单点堆积

- AI 默认不得把新增需求持续堆进单个大文件。
- 当一个文件同时承担协议处理、业务编排、数据转换、状态管理和视图渲染时，应主动拆分。
- 文件变大不是“正常演进”的默认结果，而应视为边界失守信号。

### 6. 性能基线优先于功能堆叠

- AI 不应为了快速交付而接受明显低效的实现。
- 任何新增逻辑都应至少检查：是否引入重复扫描、重复请求、重复序列化、无边界轮询或无必要的大对象拷贝。
- 性能优化不要求过早微调，但必须避免一眼可见的低效路径。

## 二、`services/core` 的依赖策略

### 核心常驻依赖

`services/core` 的核心运行依赖应长期收敛在以下范围：

- `fastapi`
- `pydantic`
- `httpx`

开发依赖允许包含：

- `pytest`
- `uvicorn`

可选依赖允许包含：

- `mempalace`
- `chromadb`
- `pypdf`

### 明确约束

- 不应把 `mempalace`、`chromadb`、`pypdf` 变回核心硬依赖。
- 不应在没有充分理由的前提下新增 ORM、任务队列、DI 容器、复杂配置框架。
- 不应为了“更工程化”而引入会显著拉长调用链的框架。

### 依赖设计要求

- 核心路径应在只安装最小依赖时仍可启动和工作。
- 可选依赖必须允许缺失，并提供明确降级行为。
- 代码中实际使用的依赖，必须在依赖声明中准确体现。
- 依赖声明变更时，应同步检查启动脚本、README 和测试环境说明。

## 三、目录职责约束

### `services/core/app/api`

- 负责 HTTP/WebSocket 协议层。
- 负责参数校验、请求解析、响应封装、错误映射。
- 不应承载过重的核心业务规则。

### `services/core/app/domain`

- 负责核心模型和稳定领域概念。
- 优先保持纯净，不直接依赖 FastAPI、外部 HTTP 客户端或可选基础设施。

### `services/core/app/*/service.py`

- 负责业务编排。
- 可以协调 repository、gateway、runtime state。
- 不应把大量协议层细节和存储细节混进核心决策逻辑。

### `services/core/app/*/repository.py`

- 负责持久化和读取。
- 默认优先文件实现、标准库方案或轻量实现。
- 如果依赖可选基础设施，必须封装在 repository 或 adapter 边界之后。

### `services/core/app/memory`

- 允许容纳 `mempalace/chromadb` 等可选能力。
- 这些能力必须被隔离在边缘模块，不应向核心域逻辑扩散。
- 缺失可选依赖时，必须有合理降级，而不是让服务直接不可用。

## 四、允许和不允许的技术选择

### 推荐做法

- 使用普通类、函数、`Protocol`、明确构造参数来表达依赖关系。
- 使用 `Pydantic` 做输入输出和结构化模型。
- 使用 `httpx` 调用外部模型或 HTTP 服务。
- 使用文件存储或 `sqlite3` 处理轻量持久化需求。

### 不推荐做法

- 引入 `sqlalchemy`、`sqlmodel`、`beanie` 一类 ORM 作为默认方案。
- 引入 `celery`、`rq`、`dramatiq` 一类队列系统作为默认后台机制。
- 引入 `dependency_injector` 一类容器来管理当前规模的服务对象。
- 引入大型配置框架替代当前明确的配置函数。
- 为了抽象而抽象，创建过多一层转一层的包装类。

## 五、AI 修改代码时的工作方式

### 1. 先读上下文，再下手

在修改任何功能前，至少应先确认：

- 入口文件在哪里。
- 相关模型在哪里定义。
- 真实依赖链经过哪些 service / repository / adapter。
- 是否已有测试覆盖该路径。

如果还没读清楚这些，就不应急于重写。

### 2. 优先沿现有模式扩展

- 如果仓库已有清晰模式，优先复用。
- 如果同类功能已有实现，优先对齐而不是另起一套。
- 如果现有结构虽不完美但可理解，优先局部修补。

### 3. 控制改动半径

- 单次改动应尽量集中在同一条行为链上。
- 如果一个需求会牵出多个方向，应先实现最小闭环。
- 不要在同一次提交里顺手做无关清理和风格漂移。

### 4. 保留降级与回退能力

- 可选能力要保证缺失时的可运行性。
- 高风险流程要保留清晰的失败路径和错误信息。
- 不要为了“自动化体验更好”去偷掉人工确认、权限边界或安全措施。

### 5. 强制控制文件和函数体积

以下阈值默认适用于日常开发；如确有必要超出，必须在改动说明中明确理由。

#### 文件体积阈值

- 后端 Python 业务文件，目标控制在 300 行以内。
- 后端 Python 业务文件，超过 500 行应优先拆分。
- 后端 Python 业务文件，超过 800 行时，除启动编排、测试样例或极少数聚合文件外，默认视为需要治理。
- 前端页面或复杂容器组件，目标控制在 250 行以内。
- 前端页面或复杂容器组件，超过 400 行应优先拆分。
- 前端页面或复杂容器组件，超过 600 行时默认不继续往内堆新逻辑，应先拆分再扩展。
- 单个工具类、适配器、网关文件如果超过 400 行，应检查是否同时承担了协议、转换、缓存、重试、错误映射等多种职责。

#### 函数体积阈值

- 普通函数目标控制在 40 行以内。
- 超过 60 行的函数应检查是否混入了多个阶段：校验、查询、计算、落盘、组装响应。
- 超过 100 行的函数默认应拆分，除非它本质上是在表达一个线性且不可再切的流程。

#### 拆分优先级

- 先按职责拆：协议层、编排层、存储层、转换层、视图层分开。
- 再按领域拆：chat、memory、goals、orchestrator 等行为链分开。
- 最后才按“工具函数”拆；不要把有业务含义的逻辑都丢进 `utils`。

### 6. 重复代码治理规则

- 相同或近似逻辑出现第 2 次时，就应判断是否值得抽出公共实现。
- 只有在两个以上真实使用点都成立时，才引入抽象；不要为了预防未来重复而提前过度封装。
- 提取公共逻辑时，优先抽“稳定步骤”而不是抽“可变流程”。
- 如果抽象后的命名比复制两段代码更难理解，则暂不抽象。
- 前端不应复制同一套请求状态、错误处理、格式化和渲染分支；后端不应复制同一套校验、仓储读写和错误映射。

### 7. 明确的性能禁区

AI 默认不应提交以下实现，除非已明确验证成本可接受：

- 在循环内重复发起 HTTP 请求、文件读取、数据库读取或模型调用。
- 对同一批数据反复做全量扫描、排序、序列化或反序列化。
- 在前端渲染过程中直接执行高成本计算、复杂过滤或大对象构造。
- 用全局大状态驱动局部小更新，导致无关组件频繁重渲染。
- 在热路径中添加无上限日志、无上限历史拼接或无上限内存缓存。
- 以轮询替代事件或状态驱动，且没有频率上限、退出条件和失败退避。

### 8. 默认性能做法

- 能复用一次查询结果时，不重复查询。
- 能增量更新时，不全量重建。
- 能在边界处做一次数据转换时，不在链路中间重复转换。
- 能分页、限量、截断时，不默认返回整批数据。
- 前端优先缩小状态影响范围，后端优先缩短热路径调用链。

### 9. 已有超大文件的处理原则

- 仓库里已经偏大的文件，不代表可以继续往里堆逻辑。
- 对超标文件的新增改动，优先采用“借机拆一层”的方式治理。
- 治理应以低风险、小步方式进行：先提取纯函数、子组件、转换器、仓储接口，再考虑更大重构。
- 不要求一次性把历史债务清空，但不允许继续恶化。

## 六、文档与测试要求

### 文档要求

出现以下情况时，应同步更新文档：

- 依赖策略变化。
- 启动方式变化。
- 配置项变化。
- 模块职责变化。
- 可选能力的启用方式或降级行为变化。

### 测试要求

优先补以下类型的测试：

- API 合约测试。
- repository 行为测试。
- 降级路径测试。
- 启动与健康检查路径测试。

如果本地无法运行测试，必须明确说明：

- 哪些测试没跑。
- 为什么没跑。
- 主要剩余风险是什么。

### 提交前自检

AI 在完成改动前，至少应自检以下问题：

- 是否把本应拆开的职责继续塞进了同一文件。
- 是否让单个函数承担了过多阶段性工作。
- 是否复制了已有逻辑，而不是复用或抽取稳定公共部分。
- 是否引入了明显的重复请求、重复扫描或重复渲染。
- 是否让调用链更长了，但没有换来清晰收益。
- 是否给后续接手者留下了清楚的模块边界和测试入口。

如需快速检查当前仓库中的超预算业务文件，可运行：

```bash
python3 tools/check_file_budgets.py
```

## 七、AI 不应擅自做的事

- 不应擅自引入新架构层。
- 不应擅自扩大自动执行权限。
- 不应擅自把可选依赖升级为硬依赖。
- 不应擅自改动产品定位，让系统更偏“工具平台”而非“数字人本体”。
- 不应在不了解全链路的情况下做大范围命名迁移或目录迁移。
- 不应把“为了让代码更漂亮”置于“让系统更稳定、更可接手”之前。
- 不应默认接受“大文件继续变大”作为实现路径。
- 不应为了少拆文件而把多个职责硬塞进单一 service、route、page 或 component。
- 不应为了图省事复制整段旧逻辑再做局部改写。
- 不应提交明知存在明显性能浪费的热路径实现。

## 八、评审清单

每次重要改动，AI 或人类评审者至少应回答以下问题：

- 这次改动解决的核心问题是什么。
- 这次改动是否保持了数字人本体优先。
- 这次改动是否增加了隐式复杂度。
- 这次改动是否让任何核心文件继续膨胀。
- 这次改动是否制造了新的重复逻辑。
- 这次改动是否引入了明显可避免的性能浪费。
- 这次改动是否引入了新的硬依赖。
- 这次改动在缺少可选依赖时是否仍然可运行。
- 后续接手者是否能在较短时间内追踪主要行为链。
- 是否需要同步补文档或测试。

## 九、默认判断规则

当多个方案都可行时，优先选择同时满足以下条件的方案：

- 依赖更少。
- 调用链更短。
- 目录职责更清楚。
- 文件体积更可控。
- 重复逻辑更少。
- 热路径更轻。
- 降级路径更明确。
- 更容易被后续 AI 和人类快速理解。

如果一个方案看起来更“高级”，但会明显增加理解成本、初始化复杂度、运行时魔法或维护门槛，则默认不选。

## 十、结论

`xiao_yan` 项目，尤其是 `services/core`，应长期坚持轻依赖、显式结构、可选能力边缘化的策略。

对 AI 最友好的项目，不是“用了最多流行库”的项目，而是“边界稳定、路径清楚、行为可验证”的项目。后续所有接手、修复、扩展与重构，都应以此为默认准则。
