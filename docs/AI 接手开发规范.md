# AI 接手开发规范

## 目的

本文档用于约束 AI 在 `xiao_yan` 项目中的接手、续写、重构与修复行为。

目标不是让 AI “尽量多做事”，而是让 AI 在理解项目目标、架构边界和依赖策略的前提下，稳定地做对事、少做错事、少引入长期维护负担。

本文档默认适用于整个仓库，重点适用于 `services/core`。

## 核心目标

- 优先保护“小晏”作为持续存在数字人的主体性、连续性和边界感。
- 优先保持代码可读、可追踪、可回退，而不是追求表面上的先进或复杂。
- 优先让后续 AI 和人类都能快速接手，而不是制造只对当前修改者友好的局部优化。

## 一、总原则

### 1. 显式优先于隐式

- 优先使用普通 Python 模块、明确函数调用和清晰数据流。
- 避免依赖大量框架魔法、运行时注册、隐式注入和难追踪的全局副作用。
- 一个改动应尽量让人通过 `rg` 和目录结构就能追到主要行为。

### 2. 少依赖优先于多依赖

- 新功能默认先用标准库和现有依赖实现。
- 只有在现有手段明显不够、并且收益长期成立时，才考虑新增第三方库。
- 引入新库前必须先回答：
  - 不用它能不能做。
  - 它会不会引入新的隐式行为或复杂初始化链路。
  - 它是否会降低后续 AI 的接手效率。

### 3. 小步修改优先于大重写

- 优先做局部、可验证、可回滚的变更。
- 没有明确收益时，不做跨模块的大范围重排。
- 已有接口如果仍然清楚可用，优先在现有结构上扩展，而不是先重构再实现。

### 4. 边界优先于技巧

- 先保证模块职责清楚，再考虑抽象是否优雅。
- 避免为了“看起来更通用”而提前引入复杂抽象层。
- 如果一个抽象没有同时减少重复、降低耦合、提升可读性，就不应引入。

## 二、`services/core` 的依赖策略

### 核心常驻依赖

`services/core` 的核心运行依赖应长期收敛在以下范围：

- `fastapi`
- `pydantic`
- `httpx`

开发依赖允许包含：

- `pytest`
- `uvicorn`

可选依赖允许包含：

- `mempalace`
- `chromadb`
- `pypdf`

### 明确约束

- 不应把 `mempalace`、`chromadb`、`pypdf` 变回核心硬依赖。
- 不应在没有充分理由的前提下新增 ORM、任务队列、DI 容器、复杂配置框架。
- 不应为了“更工程化”而引入会显著拉长调用链的框架。

### 依赖设计要求

- 核心路径应在只安装最小依赖时仍可启动和工作。
- 可选依赖必须允许缺失，并提供明确降级行为。
- 代码中实际使用的依赖，必须在依赖声明中准确体现。
- 依赖声明变更时，应同步检查启动脚本、README 和测试环境说明。

## 三、目录职责约束

### `services/core/app/api`

- 负责 HTTP/WebSocket 协议层。
- 负责参数校验、请求解析、响应封装、错误映射。
- 不应承载过重的核心业务规则。

### `services/core/app/domain`

- 负责核心模型和稳定领域概念。
- 优先保持纯净，不直接依赖 FastAPI、外部 HTTP 客户端或可选基础设施。

### `services/core/app/*/service.py`

- 负责业务编排。
- 可以协调 repository、gateway、runtime state。
- 不应把大量协议层细节和存储细节混进核心决策逻辑。

### `services/core/app/*/repository.py`

- 负责持久化和读取。
- 默认优先文件实现、标准库方案或轻量实现。
- 如果依赖可选基础设施，必须封装在 repository 或 adapter 边界之后。

### `services/core/app/memory`

- 允许容纳 `mempalace/chromadb` 等可选能力。
- 这些能力必须被隔离在边缘模块，不应向核心域逻辑扩散。
- 缺失可选依赖时，必须有合理降级，而不是让服务直接不可用。

## 四、允许和不允许的技术选择

### 推荐做法

- 使用普通类、函数、`Protocol`、明确构造参数来表达依赖关系。
- 使用 `Pydantic` 做输入输出和结构化模型。
- 使用 `httpx` 调用外部模型或 HTTP 服务。
- 使用文件存储或 `sqlite3` 处理轻量持久化需求。

### 不推荐做法

- 引入 `sqlalchemy`、`sqlmodel`、`beanie` 一类 ORM 作为默认方案。
- 引入 `celery`、`rq`、`dramatiq` 一类队列系统作为默认后台机制。
- 引入 `dependency_injector` 一类容器来管理当前规模的服务对象。
- 引入大型配置框架替代当前明确的配置函数。
- 为了抽象而抽象，创建过多一层转一层的包装类。

## 五、AI 修改代码时的工作方式

### 1. 先读上下文，再下手

在修改任何功能前，至少应先确认：

- 入口文件在哪里。
- 相关模型在哪里定义。
- 真实依赖链经过哪些 service / repository / adapter。
- 是否已有测试覆盖该路径。

如果还没读清楚这些，就不应急于重写。

### 2. 优先沿现有模式扩展

- 如果仓库已有清晰模式，优先复用。
- 如果同类功能已有实现，优先对齐而不是另起一套。
- 如果现有结构虽不完美但可理解，优先局部修补。

### 3. 控制改动半径

- 单次改动应尽量集中在同一条行为链上。
- 如果一个需求会牵出多个方向，应先实现最小闭环。
- 不要在同一次提交里顺手做无关清理和风格漂移。

### 4. 保留降级与回退能力

- 可选能力要保证缺失时的可运行性。
- 高风险流程要保留清晰的失败路径和错误信息。
- 不要为了“自动化体验更好”去偷掉人工确认、权限边界或安全措施。

## 六、文档与测试要求

### 文档要求

出现以下情况时，应同步更新文档：

- 依赖策略变化。
- 启动方式变化。
- 配置项变化。
- 模块职责变化。
- 可选能力的启用方式或降级行为变化。

### 测试要求

优先补以下类型的测试：

- API 合约测试。
- repository 行为测试。
- 降级路径测试。
- 启动与健康检查路径测试。

如果本地无法运行测试，必须明确说明：

- 哪些测试没跑。
- 为什么没跑。
- 主要剩余风险是什么。

## 七、AI 不应擅自做的事

- 不应擅自引入新架构层。
- 不应擅自扩大自动执行权限。
- 不应擅自把可选依赖升级为硬依赖。
- 不应擅自改动产品定位，让系统更偏“工具平台”而非“数字人本体”。
- 不应在不了解全链路的情况下做大范围命名迁移或目录迁移。
- 不应把“为了让代码更漂亮”置于“让系统更稳定、更可接手”之前。

## 八、评审清单

每次重要改动，AI 或人类评审者至少应回答以下问题：

- 这次改动解决的核心问题是什么。
- 这次改动是否保持了数字人本体优先。
- 这次改动是否增加了隐式复杂度。
- 这次改动是否引入了新的硬依赖。
- 这次改动在缺少可选依赖时是否仍然可运行。
- 后续接手者是否能在较短时间内追踪主要行为链。
- 是否需要同步补文档或测试。

## 九、默认判断规则

当多个方案都可行时，优先选择同时满足以下条件的方案：

- 依赖更少。
- 调用链更短。
- 目录职责更清楚。
- 降级路径更明确。
- 更容易被后续 AI 和人类快速理解。

如果一个方案看起来更“高级”，但会明显增加理解成本、初始化复杂度、运行时魔法或维护门槛，则默认不选。

## 十、结论

`xiao_yan` 项目，尤其是 `services/core`，应长期坚持轻依赖、显式结构、可选能力边缘化的策略。

对 AI 最友好的项目，不是“用了最多流行库”的项目，而是“边界稳定、路径清楚、行为可验证”的项目。后续所有接手、修复、扩展与重构，都应以此为默认准则。
