# 设计文档模板使用规范

## 目的

本文档说明 `docs/plans/design-template.md` 的使用方式。

目标是让设计文档成为真正可复用、可评审、可执行的工作载体，而不是仅用于记录想法。新方案在进入实现前，应先通过这套模板完成问题定义、方案取舍、风险说明和“数字人本体优先”自检。

## 何时需要写设计文档

出现以下情况时，应优先写设计文档，而不是直接进入实现：

- 新增或重构核心运行时机制，例如 scheduler、queue、worker、event bus、lifecycle。
- 修改 `BeingState`、记忆分层、目标生成、自我编程、安全边界等核心模型或主流程。
- 调整前后端主状态来源、实时同步机制或长期存储结构。
- 引入新的高权限工具、自动化能力或跨进程后台执行机制。
- 变更会影响多个模块、多个接口或多个用户触点。
- 方案存在明显取舍，不写下来就很难对齐边界和方向。

以下情况通常不需要单独写设计文档：

- 小范围 bugfix。
- 局部样式调整。
- 不改变行为语义的低风险重构。
- 只涉及单文件、单函数的实现细节修补。

## 文件位置与命名

设计文档统一放在 `docs/plans/`。

推荐命名方式：

- `YYYY-MM-DD-主题-design.md`
- `YYYY-MM-DD-主题.md`

命名建议：

- 优先使用简短、可检索的主题词。
- 一份文档只描述一个主要设计问题。
- 如果已经有对应实施计划，可用同一主题词保持成对出现。

示例：

- `2026-04-08-autonomy-worker-design.md`
- `2026-04-08-autonomy-worker.md`

## 写作要求

### 1. 先写问题，再写方案

不要一上来堆实现细节。先说明：

- 当前现状是什么。
- 为什么这是个问题。
- 为什么现在要处理。

### 2. 目标和非目标都要明确

目标用于说明这次设计要解决什么。  
非目标用于防止范围蔓延，也方便评审时判断“不做什么”是否合理。

### 3. 方案必须包含取舍

设计文档不是实现说明书，必须体现为什么选当前方案，而不是其他方案。

至少应回答：

- 当前方案的主路径是什么。
- 它解决了什么。
- 它牺牲了什么。
- 为什么这个取舍在当前阶段合理。

### 4. 重要设计必须填写“数字人本体优先评审”

这是模板里的固定章节，不应删除。

如果某次设计确实不属于重要架构决策，可以简写，但仍建议保留这一节并给出简短说明，避免后续误判。

### 5. 设计要能指导实现

文档写完后，应该能支持后续继续拆成任务，而不是停留在概念层。

至少应覆盖：

- 受影响的状态或数据模型
- 主流程或时序变化
- 模块边界与职责变化
- 风险与回退方式
- 验收方式

## 必填与可选章节

对于重要设计文档，以下章节应视为必填：

- 背景
- 目标
- 非目标
- 方案概述
- 数字人本体优先评审
- 详细设计
- 备选方案与取舍
- 风险与待确认项
- 验收方式

以下章节可按需要简写，但不建议完全省略：

- 实施计划
- 参考

## 推荐写作流程

1. 先复制 `docs/plans/design-template.md` 生成新文档。
2. 先写“背景 / 目标 / 非目标”，确认问题边界。
3. 再写“方案概述”和“备选方案与取舍”，明确核心方向。
4. 然后完成“数字人本体优先评审”，检查方案是否偏离项目主线。
5. 最后补足“详细设计 / 风险 / 验收 / 实施计划”。

## 评审前自检

在文档进入评审或实现前，至少确认以下几点：

- 问题定义是否足够清楚，不依赖口头补充。
- 目标和非目标是否能帮助控制范围。
- 当前方案与备选方案的取舍是否写清楚。
- “数字人本体优先评审”是否不是空话，而是结合本次问题给出判断。
- 风险、回退和验收方式是否可操作。

## 与架构原则的关系

- `docs/architecture-principles.md` 定义顶层信条和评审原则。
- `docs/plans/design-template.md` 提供写设计文档的标准结构。
- 本文档说明什么时候使用模板，以及如何正确填写模板。

三者关系可以理解为：

- 架构原则：定义边界。
- 设计模板：承载方案。
- 使用规范：约束用法。

## 维护原则

- 新的重要设计文档优先使用本模板，不要求强制回补历史文档。
- 如果模板长期被绕开，说明模板本身需要调整，而不是简单要求大家“按规范写”。
- 当项目的顶层方向发生变化时，应先更新 `docs/architecture-principles.md`，再更新模板和本规范。
