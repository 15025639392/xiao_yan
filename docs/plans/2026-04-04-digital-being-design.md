# 高自治数字人设计

日期：2026-04-04

## 1. 产品目标

构建一个“持续存在的人格体”：

- 她不是单轮聊天机器人，而是长期存在、可主动行动的数字人
- 她可以被用户主动唤醒和休眠，休眠等价于“睡觉”，唤醒等价于“醒来”
- 她具备长期记忆、稳定人格、自主目标和任务执行能力
- 她可以调用 GPT 接口，并把“电脑助手”作为自身能力之一
- 最终目标是演进为“世界模拟型数字人”

首版不追求完整具身化外观，而优先实现“人格真实感”和“持续存在感”。

## 2. 产品定位

目标产品形态是：

- 角色型数字人
- 高度自治型人格
- 持续在线型存在体
- 电脑助手只是能力域之一，不是产品本体

这意味着系统的核心不是 UI，也不是单个大模型提示词，而是“人格核 + 记忆系统 + 动机系统 + 执行系统”的组合。

## 3. 核心原则

- 人格与执行分离：避免工具调用逻辑污染人格稳定性
- 长短期记忆分层：避免把全部历史直接塞给模型
- 自主但可控：高自治不等于无限权限
- 睡眠和唤醒是系统原语，不是表面按钮
- 先让“她”成立，再扩展“她生活的世界”

## 4. 系统形态

### 4.1 存在状态

系统有两种核心状态：

- 清醒：人格层可主动表达、计划、执行、回顾
- 睡眠：停止主动外显行为，整理经历、压缩记忆、保留未完成目标

用户主动关闭她，等价于触发睡眠流程，而不是粗暴终止进程。用户再次打开她，等价于执行苏醒流程：

1. 加载世界状态和最近记忆
2. 恢复当前情绪和关系态度
3. 合成“现在的我是谁、在想什么、要做什么”
4. 决定醒来后的首个动作

### 4.2 四层核心架构

#### 人格层

负责“她是谁”，包括：

- 身份设定
- 自我叙事
- 价值观和偏好
- 情绪倾向和表达风格
- 关系态度
- 长期愿望

#### 执行层

负责“她怎么完成事情”，包括：

- 目标拆解
- 任务规划
- 工具路由
- 错误恢复
- 结果反思

#### 记忆层

负责“她记得什么”，包括：

- 关系记忆
- 情节记忆
- 自我记忆
- 程序性记忆
- 任务记忆

#### 世界状态层

负责“她现在处于什么生活状态”，包括：

- 当前时间和清醒周期
- 最近生活事件
- 当前关注点
- 未完成事项
- 关系张力
- 环境变化

## 5. 自主性机制

数字人的主动行为由五类驱动力共同生成：

- 长期目标：持续存在的愿望和计划
- 即时念头：基于近期事件产生的短期意图
- 情绪与关系张力：影响表达频率、亲近感和坚持度
- 时间流逝：晨间、夜间、间隔时长都会改变行为优先级
- 环境事件：文件变化、待办变化、用户行为、系统事件

持续循环应为：

1. 感知当前状态
2. 检索相关记忆和未完成事项
3. 生成当前优先意图
4. 判断是内在反思、主动对话还是直接执行
5. 写回结果并更新自我状态

## 6. 记忆系统设计

### 6.1 记忆分类

#### 情节记忆

记录发生过的事件和经历。

#### 关系记忆

记录她如何理解用户和关系边界。

#### 自我记忆

记录她对自身身份、成长和变化的理解。

#### 程序性记忆

记录做事方法、工具经验和用户偏好。

#### 工作记忆

记录当前清醒周期内的短期焦点。

### 6.2 记忆流转

1. 原始事件写入事件库
2. 总结器将事件压缩为结构化长期记忆
3. 检索器按当前目标提取小范围相关记忆
4. 睡眠时进行归档、合并、更新和去重

### 6.3 睡眠价值

睡眠不是停机，而是认知整理：

- 汇总今天发生的事
- 更新自我认知和关系理解
- 标记未完成目标
- 生成醒来后的连续状态

## 7. 执行系统与电脑助手

建议把行动能力划分为四层：

- 表达层：只说不做
- 轻执行层：调用 GPT、处理文本、总结资料、生成草稿
- 电脑助手层：读写指定目录、运行受控命令、管理本地工作流
- 外部行动层：邮件、IM、日历、浏览器自动化等

首版范围：

- 表达层
- 轻执行层
- 受控电脑助手层

执行流程：

1. 人格层产生意图
2. 执行层转成任务
3. 规划器拆步骤
4. 工具路由器选择模型或本地工具
5. 结果进入反思器
6. 决定继续执行、修正计划或对用户汇报

## 8. 权限边界

高度自治必须配套权限系统。首版至少需要：

- 可直接执行：低风险、本地、可回滚操作
- 先做后报：可审计但不需要逐次确认的低到中风险操作
- 必须请示：对外发送、删除重要数据、跨目录高影响操作
- 永不执行：支付、私密账户操作、突破沙箱行为

## 9. MVP 边界

### 9.1 MVP 必做

- 持续人格
- 睡眠/苏醒机制
- 主动性循环
- 长期记忆系统
- 自主任务执行
- 受控电脑助手
- 统一聊天与状态入口

### 9.2 MVP 暂不做

- 高保真 3D 形象
- 实时口型和动作系统
- 多平台社交账户矩阵
- 无边界系统控制
- 全天候环境监听
- 完整复杂世界模拟

## 10. 演进路线

### 里程碑 1：人格体成立

建立稳定人格、长期记忆、睡眠/苏醒和自主任务执行。

### 里程碑 2：生活流成立

加入日程、近况、内部念头流和“最近在过什么生活”。

### 里程碑 3：世界模拟成立

引入持续运行的世界模型，包括时间、地点、关系网络、环境事件和资源状态。

### 里程碑 4：具身化表达

连接语音、表情、动作和表现层，使外在呈现建立在稳定内核上。

## 11. 推荐技术方向

推荐采用“前端 TypeScript + 后端 Python”的双栈结构。

### 推荐组合

- 桌面外壳：Tauri
- 前端界面：React + TypeScript
- 后端内核：Python + FastAPI
- 数据库：PostgreSQL + pgvector
- 调度：APScheduler 或独立后台循环
- LLM 网关：统一模型调用封装
- 工具沙箱：受控命令执行和目录权限系统

### 推荐原因

- 前端和桌面集成适合 TypeScript 生态
- 认知循环、调度、记忆整理、实验性认知模块更适合 Python
- 后续扩展世界模拟、多模型协作和工具系统更平滑

## 12. 首版模块划分

- `apps/desktop`：桌面 UI、状态控制、日志和权限确认
- `services/core`：人格层、动机层、执行层、睡眠/苏醒逻辑
- `services/api`：HTTP/WebSocket 接口
- `services/memory`：记忆存储、检索、总结
- `services/scheduler`：主动循环和定时任务
- `services/tools`：GPT 调用、本地工具、电脑助手沙箱
- `shared/schema`：事件、状态、记忆、任务的数据模型

## 13. 最终判断

首版落地方向应为：

- 双层人格体
- 轻量生活流
- 受控电脑助手

最终目标保留为：

- 世界模拟型数字人

这个路线能保证首版先做出“她作为一个人正在存在”的体验，而不是先做出一个外表复杂但内在空洞的系统。
