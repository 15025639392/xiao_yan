# Large File Split Advisor

在用户要修改已经偏大、偏热或职责混杂的文件时使用本 skill。

尤其适合：

- 后端业务文件超过 500 行
- 后端业务文件接近或超过 800 行
- 前端页面或容器组件超过 400 行
- 函数超过 60 行且明显混合职责
- route、service、component 同时承担协议、编排、存储、转换、展示等多种职责

## 目标

不是为了“把文件拆小而拆小”，而是找出最小、最稳、最可回退的拆分切口。

本 skill 优先回答：

- 这次改动还能不能局部加？
- 如果不能，最小承载单元应该是什么？
- 拆分是提 helper、提 module，还是先停下来出设计？

## 必读上下文

开始前先读：

- `docs/AI 接手开发规范.md`
- 当前目标文件
- 与目标文件直接相关的入口、调用方和测试

## 默认工作流

1. 先确认用户这次真正要改的行为链，不要把历史债务全部打包处理。
2. 检查目标文件的体积、热点程度和职责分布。
3. 标出当前文件中的职责块，优先按这几类识别：
   - 协议处理
   - 业务编排
   - 存储读写
   - 数据转换
   - UI 展示
   - 状态同步
4. 判断新增改动是否会继续给该文件增加新职责。
5. 给出最小拆分策略：
   - `keep_stable`：这轮先不拆，局部改动即可
   - `extract_helper`：提取纯函数或纯展示子块
   - `extract_module`：提取一个明确职责的小模块
   - `split_before_change`：不拆会继续恶化，先拆再改
6. 如果建议拆分，必须明确：
   - 拆什么职责
   - 新文件放哪里
   - 原文件保留什么
   - 哪些测试和文档要跟着动
7. 控制拆分范围，只服务当前需求，不顺手做整片重构。

## 必须产出

- `文件现状`
- `职责分布`
- `拆分结论`
- `建议承载点`
- `改动顺序`
- `同步项`

推荐格式：

- `文件现状: 781 行 goals admission 文件，已混合 admission 规则、队列策略和统计输出`
- `职责分布: admission decision + defer persistence + stats assembly`
- `拆分结论: extract_module`
- `建议承载点: 提取 defer queue adapter 到独立模块`
- `改动顺序: 先提取模块，再迁移调用，再补测试`
- `同步项: tests + docs/plans + file budget check`

## 判断规则

- 如果只是大，但职责仍单一，可暂时不拆。
- 如果这次改动会把文件从“偏大”推向“更混合”，优先拆一个最小职责层。
- 如果纯逻辑已经能独立命名，优先提 helper 或 module，而不是继续内联。
- 如果拆分需要跨很多层同时改动，先缩小目标，不要借机大重构。

## 仓库内特别关注

- `services/core/app/api/*.py`：不要把更多核心业务规则塞进 route
- `services/core/app/*/service.py`：不要同时承担决策、存储细节和协议映射
- `apps/desktop/src/App.tsx` 和大容器组件：优先分离容器状态与展示片段
- 已经过预算的文件，这轮若继续加新方向职责，默认视为拆分候选

## 强约束

- 不要为了追求“更优雅”而提出大规模重排。
- 不要把历史债务清理和当前需求绑成一个不可回退的大改动。
- 不要抽象出空壳 `utils` 作为默认拆分落点。
- 如果决定不拆，也要说明为什么这轮继续局部修改是安全的。
