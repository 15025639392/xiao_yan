# Docs And Tests Sync Guard

在任何会改变“别人如何理解或验证系统”的改动后使用本 skill。

尤其适合：

- 改依赖声明
- 改配置项、环境变量、启动脚本
- 改模块职责或目录边界
- 改降级行为、可选依赖行为
- 改 API、状态字段、数据文件结构
- 改 tools/skills
- 改默认流程、主链路、验证方法

## 目标

确保这轮改动结束后：

- 文档没有误导
- 测试没有漏补
- 验证命令可复现
- 后来接手的人能看懂发生了什么

## 低 Token 默认做法

项目初期做收尾同步时，优先只检查和本轮改动直接相关的同步项。

默认只看：

- 本轮实际改过的文件
- 同域测试
- 直接相关文档
- 直接相关 skill 文档
- 当前实际跑过的验证命令

默认不做：

- 全量扫 `docs/`
- 全量扫所有 runbook
- 把无关模块的旧说明一起清理

如果只是局部 bug 修复或单文件改动，优先给最小同步范围，而不是扩成整仓文档盘点。

## 默认工作流

1. 列出本轮改动类型。
   - 代码逻辑
   - 配置 / 依赖
   - 文档 / skill
   - 状态 / API / 数据结构
2. 反推需要同步的产物：
   - 测试
   - 设计文档 / plans
   - runbook
   - 启动说明
   - 与该主题相关的 skill 文档
3. 用搜索确认旧说明是否仍然存在。
   - 优先用 `rg` 搜关键术语、路径、环境变量、接口名、状态字段
4. 判断同步状态：
   - `already_aligned`
   - `needs_test_update`
   - `needs_doc_update`
   - `needs_both`
5. 如果这轮改动改变了结构、默认入口、依赖、模块职责或降级行为，补充或更新落盘文档。
6. 收尾时明确实际跑过哪些测试和检查，没跑的也要写出来。

如果用户只需要一个最小收尾结论，可以压缩为：

1. 列本轮改动类型
2. 列本轮需要同步的最小文档/测试范围
3. 写实际验证命令
4. 写剩余风险

## 必须产出

- `同步范围`
- `文档状态`
- `测试状态`
- `验证命令`
- `剩余风险`

推荐格式：

- `同步范围: tests + docs/plans + skill 文档`
- `文档状态: needs_doc_update`
- `测试状态: needs_test_update`
- `验证命令: pytest ... ; python3 tools/check_file_budgets.py`
- `剩余风险: 未覆盖手工交互路径`

## 文档同步提示

优先检查：

- `docs/`
- `docs/plans/`
- `docs/runbooks/`
- `tools/skills/`
- 与改动直接相关的注释、脚本说明、验证命令

如果改动影响 AI 接手方式、运行规则或能力边界，也要检查：

- `AGENTS.md`
- `docs/AI 接手开发规范.md`
- `docs/architecture-principles.md`
- `docs/testing-strategy.md`
- `docs/test-checklist-template.md`

## 测试同步提示

- 先找现有同域测试，优先补在现有测试文件附近
- 如果是 bug 修复，至少补一个能覆盖这次失效分支的断言
- 如果是纯文档/skill 改动，可以不强行新增测试，但要说明为何不需要

## 强约束

- 不要只在最终回复里口头说“已同步”，要真的搜索并确认。
- 不要改了配置、依赖、状态结构后还保留旧验证命令。
- 不要把“之后再补文档/测试”当默认选项。
- 如果没跑测试，要明确未验证的链路和风险。
